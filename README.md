# Authentication

Email + password authentication with JWT tokens. The Auth module is separate from the System module — it only owns credentials (`passwordHash`), while System owns user identity (`email`, `name`, `roles`, `status`). This makes it possible to swap the Auth module for an external identity provider (e.g. Keycloak) without touching the rest of the system.

## Tokens

- **Access token** — short-lived JWT (15m), sent as `Authorization: Bearer <token>` on every request.
- **Refresh token** — longer-lived JWT (7d), used to get a new access token without re-entering credentials. Stateless (not stored server-side), so logout is client-side only.
- **Activation token** — JWT (48h), generated by an admin, used once by the user to set their password.

## First Admin Bootstrap

```bash
# 1. Create admin user (UNACTIVATED)
pnpm run cli create-admin \
  --email admin@company.com \
  --first-name Admin --last-name Boss

# 2. Set password and activate
pnpm run cli activate-account \
  --user-id <uuid-from-step-1> \
  --password SecurePass123
```

The admin can now log in via the API.

## Creating New Users

```
Admin logs in
  POST /auth/login { email, password } -> { accessToken, refreshToken }

Admin creates a user
  POST /system-user { email, firstName, lastName, roles } -> { userId }

Admin generates activation token
  POST /auth/activation-token { userId } -> { activationToken }

Admin shares the activation token with the user (email, chat, etc.)

User activates their account (public, no auth required)
  POST /auth/activate { activationToken, password } -> { accessToken, refreshToken }
```

## Connecting Users to Employees

A SystemUser (login identity) and an Employee (HR record) are separate concepts. After a user activates their account, an admin links them:

```
Admin creates an employee record
  POST /employees { firstName, lastName } -> { employeeId }

Admin links the employee to their system user
  PATCH /employees/:employeeId/link-user { userId }

Admin assigns a position (which carries permissions)
  PATCH /employees/:employeeId/assign-position { positionKey }
```

Once linked, the user's permissions are resolved through HR: position defaults + per-user overrides. Users with the `administrator` system role bypass permission checks entirely.

## API Endpoints

| Method | Route                  | Auth?  | Purpose                                  |
|--------|------------------------|--------|------------------------------------------|
| POST   | `/auth/login`          | Public | Email + password -> tokens               |
| POST   | `/auth/activate`       | Public | Set password using activation token       |
| POST   | `/auth/refresh`        | Public | Refresh token -> new access token        |
| POST   | `/auth/activation-token` | Bearer | Generate activation token (admin-only) |

All other endpoints require a valid access token.
